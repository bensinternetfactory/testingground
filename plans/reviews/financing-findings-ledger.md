# Financing Findings Ledger

## Purpose

This ledger is the canonical normalized findings store for the financing audit.
Step reports may contain richer narrative, but the final consolidation must rely
on this file for deduplicated findings state.

## Ledger Rules

- one row per normalized finding
- keep the same `Finding ID` across later substeps when the same root issue is
  revisited
- mark exact duplicates as `duplicate` and point them at the canonical finding
- use `resolved-in-analysis` only when later review proves the original concern
  is not a real issue
- use `boundary-only` in the summary or pattern tag when the finding stays in
  boundary-review scope
- update `Latest-reviewed substep` whenever a finding is reconfirmed or refined

## Finding ID Format

Use this format:

`FIN-<rule-area>-<3 digit sequence>`

Examples:

- `FIN-metadata-001`
- `FIN-rsc-boundary-002`
- `FIN-bundle-003`

The sequence is global, not per step.

## Required Fields

Every ledger entry must include:

- `Finding ID`
- `Status`
- `Severity`
- `Source`
- `Rule area`
- `Pattern tag`
- `Affected components`
- `Affected routes`
- `Shared dependency impact`
- `First-seen substep`
- `Latest-reviewed substep`
- `Summary`
- `Fix direction`
- `Evidence pointer`

## Status Values

- `open`
- `accepted`
- `duplicate`
- `deferred`
- `resolved-in-analysis`
- `blocked`
